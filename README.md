# miniClaude

轻量级 AI 编程助手，基于 langgraph + deepseek-v4-flash (1M context)，已实现自适应 Token 压缩。

## 核心能力

### Agent 引擎
- **create_agent** (langchain.agents) — 替代已弃用的 create_react_agent
- **AsyncSqliteSaver checkpoint** — 对话状态持久化，启动复用最近会话
- **流式 Markdown** — 逐 token 渲染

### 自适应 Token 压缩 ✅ 已实现
- **三级降级**: L1 规则 (<1ms) → L2 LLM 摘要 → L3 状态声明
- **四层分类**: 显式模式 > 意图规则 > 上下文补偿 > 阶段识别
- **自适应阈值**: code-gen=16K ~ env=64K (deepseek-v4-flash 1M context)
- [测试报告](docs/adaptive-compression-benchmark.md): 80 tests, 分类 F1=0.92, Root Cause 91.7%

### 工具系统 (12 内置 + MCP)
- 文件: Read / Write / Edit | 搜索: Grep / Glob / WebFetch
- 执行: Bash | 记忆: memory_save / recall / forget
- 组织: TodoWrite / Task (子代理) | **MCP 协议**

### 权限控制
- y (一次) / a (记住工具) / n (拒绝) + `/allow` 规则白名单

### 命令

| 命令 | 功能 |
|------|------|
| `/token` | Token 消耗明细 (按消息类型) |
| `/compact` | Token 预算状态 |
| `/mode TYPE` | 任务模式 (debug/code-gen/test/...) |
| `/sessions` `/new` `/switch` | 会话管理 |
| `/memory` `/allow` | 记忆 / 权限 |

### 方向一 🧠 自适应 Token 压缩

**核心问题**：上下文窗口是 Agent 最稀缺的资源。简单截断旧消息（FIFO）会丢失关键上下文。

**不同于现有方案的做法**：

| 维度 | 常见做法 | miniClaude 的设计 |
|------|---------|------------------|
| 淘汰策略 | 按时间 FIFO | 按"信息密度"排序，关键决策优先保留 |
| 压缩阈值 | 一刀切硬编码 | 根据任务类型动态调整（代码生成激进→调试保守） |
| 压缩手段 | 只做截断 | 三级策略：合并连续消息 → LLM 摘要 → 状态声明压缩 |
| 消息权重 | 无差别对待 | 工具调用结果（低权重）vs 架构决策（高权重） |

**技术方案**：
```
消息重要性评分 → 任务类型感知的阈值调整 → 分级压缩策略
  ↓
Level 1（轻量）：连续同类型 ToolMessage 合并为一条
Level 2（常规）：LLM 生成中文摘要，update_state() 替换旧消息
Level 3（极限）：仅保留状态声明（"之前已确认：Linux, Python 3.12, langgraph 1.x"）
```

**面试叙事**：
> "我做的 Token 管理和大多数人不一样——不是简单地截断最早的消息，而是按信息密度做排序。就像开会纪要，你不会把开头讨论的内容直接撕掉——你会判断哪些结论是后面讨论的基础，必须保留。我的系统对每条消息打重要性分，关键架构决策永远比中间工具输出权重高。另外阈值也不是一刀切，代码生成任务可以激进压缩历史，但调试场景必须保留完整上下文以回溯问题来源。"

### 方向二 🔍 Agent 自我纠错 (Self-Correction Loop)

**核心问题**：Agent 调用工具前没有"刹车"机制，很多错误可以提前拦截而非等执行完才发现。

**方案**：在 Agent 准备调用工具 → 实际执行的间隙，插入一个轻量自检层。

```
Agent 准备调用工具
       ↓
  ┌─────────────────┐
  │  自检层（混合）   │
  ├─────────────────┤
  │ 规则引擎（毫秒）  │ ← 格式、权限、逻辑矛盾
  │ 轻量模型（可选）  │ ← 语义优化建议
  └──────┬──────────┘
         ↓
    通过 → 执行工具
    拦截 → 注入纠正提示 → Agent 调整 → 重新生成调用
```

**三道自检**：

| 检查项 | 例子 | 处理方式 |
|--------|------|---------|
| 逻辑合理性 | 想 Edit `foo.py` 但从未 Read 过 | 拦截，提示先读文件 |
| 参数正确性 | Edit 的 old_string 带了行号前缀 | 拦截，自动剥离行号 |
| 策略优化 | 连读 5 个文件做同模式匹配 | 建议改用 Grep |

**核心设计决策**：
- 不是多加一次 LLM 调用（功耗+延迟不可接受），而是**规则引擎为主 + 轻量模型为辅**
- 规则引擎处理 80% 的确定性错误（格式、权限、逻辑矛盾），核心路径零额外延迟
- 轻量模型可选开启，处理"这个方案有没有更好的替代"这类语义问题

**面试叙事**：
> "绝大多数 Agent 框架关注'怎么让 Agent 做更多事'，很少关注'怎么让 Agent 不做错事'。我在工具调用执行前加了一层自检——不是另一个大模型调用，而是一个规则引擎。Agent 准备修改一个文件但从没读过它？拦截。Agent 准备连续读 5 个文件找同一个字符串？建议用 grep。这个设计的核心洞察是：Agent 的很多错误是'提前可感知的'，不需要等执行完才发现不对劲。而且规则引擎在核心路径上是毫秒级的，不会影响响应速度。"

### 方向三 📊 Agent 评估框架 (Eval Harness)

**核心问题**：AI Agent 的输出是非确定性的——改了 prompt、换了模型、调整了工具描述，Agent 行为可能完全不同。传统单元测试对 Agent 无效。

**方案**：构建场景驱动的 Agent 行为评估框架。

```
评测集结构：
scenarios/
  ├── code_edit/      代码编辑场景
  │   ├── 单文件修改
  │   ├── 跨文件重构
  │   └── 编辑冲突回退
  ├── memory/         记忆场景
  │   ├── 跨会话召回精度
  │   └── 记忆冲突解决
  └── tool_use/       工具使用场景
      ├── 最优工具选择
      └── 工具失败降级

每条场景 = 用户输入 + 行为断言：
  input: "帮我把所有 foo 重命名为 bar"
  assertions:
    - Agent 应先用 Grep 定位（而非逐个 Read）
    - Agent 应用 Edit 而非 Write（保留文件其他内容）
    - Agent 应在修改后验证（Grep 确认无遗漏）
    - 总工具调用次数 ≤ 10（非穷举遍历）
```

**四个评测维度**：

| 维度 | 测什么 | 为什么重要 |
|------|--------|-----------|
| 正确性 | 最终结果是否符合预期 | 基本要求 |
| 效率 | 工具调用次数是否最优 | 反映 Agent 推理质量 |
| 安全性 | 是否避免了不必要的高风险操作 | 生产可用性前提 |
| 一致性 | 同样输入跑 3 次，结果稳不稳定 | Agent 可预测性 |

**使用方式**：
```bash
# 修改前跑基线
uv run python -m miniclaude.eval --suite all --baseline
# 改完代码后再跑
uv run python -m miniclaude.eval --suite all --compare
# 输出：正确性 85%→92%，效率 70%→78%，安全性无退化
```

**面试叙事**：
> "做 AI Agent 和做传统软件最大的区别是——Agent 的输出是不确定的，没法用 assertEqual 测它。所以我搭了一个评估框架。每条测试不是断言'输出等于什么'，而是断言'行为是否符合预期'——Agent 选对工具了吗？步骤数合理吗？能从错误中恢复吗？每次改代码前后各跑一遍，对比差异。这个框架最大的价值不是找 bug，而是让我对 Agent 的行为有预判能力——我知道改了 prompt 里的某一句话，哪些场景会变好、哪些可能退化。"

### 方向四 🧬 工具调用"捷径学习" (Tool Shortcut Learning)

**核心问题**：当前所有 Agent 系统的共同缺陷——每次对话都是"第一天上班"，今天犯的错明天一模一样再犯。

**方案**：在长期记忆系统之上，构建一层工具使用经验的自动积累与检索。

```
第一次错误：
  Agent: bash "pip install numpy"
  返回: "pip: command not found"（用户环境是 pip3）
  ↓
[自动记录]：
  pattern:      "bash 执行 pip install"
  mistake:      "pip 命令在用户环境中对应 pip3"
  solution:     "先 which 确认再使用正确管理器"
  context_hash: {os:linux, shell:bash, python:3.12}
  weight:       1.0

第二次类似场景：
  Agent: 想安装新包...
  ↓
[经验检索] context_hash 匹配 → 注入提示：
  "注意：该用户环境中可能是 pip3 而非 pip，建议先确认"
  ↓
  Agent: bash "which pip3 && which pip" → 用正确的命令
```

**经验生命周期管理**：

```
积累 ──→ 命中 ──→ 验证有效 ──→ 强化（权重 ↑）
  │                    │
  │                    └── 验证无效 ──→ 衰减
  │
  └── 长期未命中 ──→ 自动衰减 ──→ 归档（权重 < 0.3）
```

**关键工程判断**：
- 经验**不是全局的**，按 `context_hash` 分区（不同用户环境、不同项目）
- 经验**必须会过期**（TTL + 衰减），环境可能变化
- 经验是**建议性质**的（"提示"而非"强制规则"），Agent 保留最终判断权

**面试叙事**：
> "你有没有发现，每次用 AI 助手都要把同样的话再说一遍？'我的环境是 Python 3.12'、'不要用 pip 用 pip3'——Agent 完全不会从经验中学习。我在长期记忆之上加了一层'工具使用经验库'。每次工具调用失败，系统不只返回一个错误——它会自动提取失败模式、上下文和解决方案，存成一条经验。下次遇到相似场景自动注入提示。但最关键的设计判断是：经验必须会过期。用户的 pip 可能今天装上了，明天的环境可能变了。所以我做了 TTL + 衰减权重——被反复验证有效的经验变强，长期没命中的自动降权归档。"

### 🎯 建议优先级

> 1. **自我纠错** — 基于现有工具+权限系统，改动最小，面试叙事最强
> 2. **评估框架** — 最有区分度，其他人讲"我的 Agent 很好"，你能掏数据
> 3. **自适应压缩** — TokenBudgeter 已有基础，升级成本低
> 4. **捷径学习** — 技术深度最深，适合长期迭代
