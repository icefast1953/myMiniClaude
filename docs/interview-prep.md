**从零构建的轻量级 AI 编程助手，聚焦 Agent 上下文管理的真问题——不是简单地截断旧消息，而是按信息密度做智能压缩，并在工具执行前插入规则引擎做自检纠错。**

## 二、核心叙事线（3-5 分钟口播稿）

> 面试官："介绍一个你最有代表性的项目。"

### 问题引入（30s）

"我做的是一个轻量级 AI 编程助手。这个项目要解决的核心问题是：**Agent 的上下文窗口，到底怎么管理？**"

"现在的大模型 context window 越来越大——deepseek-v4-flash 支持 1M token。但问题是：Agent 对话是持续增长的，它会不断调用工具、产生结果、积累历史。1M 不够用是迟早的事。大多数方案是一刀切地截断最早的消息——这就像开会开得太长了，把前 20 分钟讨论直接撕掉。但那些讨论里可能有后面所有决策的基础。"

### 亮点 1：自适应 Token 压缩（90s）

"所以我做了一套自适应压缩系统。核心思路是：**不是按时间淘汰，而是按信息密度排序。**"

"具体是三级降级策略：

- **Level 1（规则压缩）**：把连续的 ToolMessage 压缩，保留前几行 + 后几行 + 统计信息，中间丢掉。因为 Agent 真正需要知道的不是完整文件内容，而是文件结构和需要修改的那几行。这一步零延迟，纯规则。
- **Level 2（LLM 语义摘要）**：当 L1 压缩后还是超限，调用模型把历史对话生成中文摘要。关键设计是——摘要不覆盖全部对话，只压缩早于最近 N 轮的部分。最近的对话保持完整。
- **Level 3（状态声明）**：极端情况下 L2 摘要都太长。这时候只保留一个极简状态声明——'已确认：Python 3.12, langgraph, Windows 环境'——就像在对话开头贴一张便签，丢了过程但保住了事实前提。"

"压缩阈值不是一刀切的。我做了个四层任务分类器：显式模式 → 意图正则 → 上下文补偿 → 阶段识别。不同任务类型用不同阈值——**代码生成任务激进压缩，调试任务保守保留。** 配合滞后机制防止任务类型来回跳动。"

"这里有个有意思的坑：压缩逻辑写好了，但就是不触发。排查发现 AsyncSqliteSaver 的 sync 版本 `get_state()` 在 async 上下文中**静默失败**——不抛异常，但返回空结果。全部改成 `aget_state()` 之后就好了。这件事的教训是：在 async-first 的框架里，不是'用 async 更好'，是'不用 async 等于没调用'。"

### 亮点 2：自我纠错设计（60s）

"压缩解决的是'上下文太多了怎么办'。但还有一个问题：**Agent 在调用工具之前没有刹车机制。**"

"我设计了一套自我纠错层，插在'Agent 决定调工具'和'工具真正执行'之间。核心是规则引擎——不在核心路径上多加 LLM 调用，那样延迟不可接受。规则分三级：
- 🔴 确定错误自动修复：Edit 参数带了行号前缀 → 自动剥离
- 🟡 逻辑可疑硬拦截：想 Edit 一个从未 Read 过的文件 → 拦截提示
- 🔵 策略优化软建议：连续读 N 个文件搜同一模式 → 追加建议用 grep"

"核心洞察是：Agent 的大量错误是**提前可感知的**——不需要等执行完才发现不对劲。规则引擎核心路径毫秒级，不影响响应速度。"

### 亮点 3：三层记忆（30s）

"另外我重新梳理了'记忆'这个概念。同一个词其实是三个问题：短期记忆用 checkpoint 持久化、Token 预算用智能压缩淘汰、长期记忆用 Agent 自主管理。三个层级，三个不同的技术方案。"

### 收尾（10s）

"技术栈是 Python + langgraph + deepseek-v4-flash。整个项目每个模块都是自己设计实现的——不是套框架模板，是理解原理后做的最小化方案。"

## 三、技术深挖问答库

### Q1: 三级降级压缩具体怎么做的？L2 和 L3 的区别？

"三级设计的核心原则是：**优先用最便宜的方法，不行再升级。**"

"L1 是纯规则，毫秒级。只处理 ToolMessage——把大文件读取结果压缩成摘要格式，同时保留 `tool_call_id` 保证 langgraph 的消息配对不会出错。零额外 LLM 成本。"

"当 L1 后还是超阈值，启动 L2+L3。它们是**同一次 LLM 调用**——要求模型同时输出自然语言摘要和极简状态声明 YAML。我分别估算加上最近消息后的总 token，选能降到阈值以下的。L2 优先。"

"L2 和 L3 的本质区别：L2 是**语义压缩**——保留了讨论过程和决策逻辑。L3 是**事实压缩**——只保留后续推理需要的事实前提，完全丢了讨论过程。L3 是最后手段，但至少比直接截断好。"

"保证不丢关键信息靠两点：只在旧消息上压缩，最近 N 轮完整保留；切分点有 `_safe_split_point()` 确保不会把 tool_calls 和对应 ToolMessage 拆开，避免框架报错。"

---

### Q2: 四层任务分类器怎么判断当前任务类型？

"四层是递进的 fallback 关系。Layer 1 最快——用户显式 `/mode debug`，100% 置信度。Layer 2 是正则意图匹配——'跑不起来了'、'报错了' → debug；'帮我写一个' → code-gen。Layer 3 是上下文补偿——当前消息太短，往前翻看之前聊的是什么。Layer 4 是阶段识别——看最近几轮的行为模式：80% 是 read/grep → 探索阶段；有 edit/write → 实现阶段。"

"防止来回切换用了**滞后机制**：任务类型必须连续多轮都判为新类型才切换，阶段同理。就像恒温器不会因为开门那一瞬间的温度变化就重启空调。"

---

### Q3: AsyncSqliteSaver 你遇到了什么问题？

"这是整个项目最隐蔽的 bug。压缩逻辑写好但从来不触发。加日志发现 `check()` 看到的 token 数一直很小。怀疑 checkpoint 没存上——但数据库里消息确实写进去了。最后追溯到 sync 版本 `get_state()` 在 async 事件循环中返回了 `None` 而不是报错。根本原因是底层用 aiosqlite，sync 方法在某些嵌套事件循环场景获取不到正确的 loop，直接静默返回空结果。"

"如果给 langgraph 提 PR：第一，sync 方法和 async 方法不应该共存于同一个类；第二，至少在 sync 方法里加 RuntimeWarning 检测自己在 async 上下文中被调用。"

---

### Q4: 自我纠错为什么用规则引擎而不是 LLM？

"首先要明确：规则引擎的定位是处理确定性错误——那些不需要语义就能判断的问题。Edit 没 Read 过文件 → 逻辑矛盾；old_string 带了行号前缀 → 格式错误。这些都是确定性的。"

"80% 的 Agent 错误是这种类型。规则引擎快、准、零误判。剩下 20% 确实需要语义理解，但多加一次 LLM 调用每次工具执行延时会显著增加——对于需要连续调多个工具的 Agent 来说，累积延迟用户能明显感知。这是取舍。"

---

### Q5: 你的三层记忆和普通 RAG 有什么区别？

"RAG 解决的是 '从大量文档里检索相关片段'，但 Agent 的记忆有三类完全不同的问题：
- 短期记忆：对话历史持久化，用 checkpoint，不是检索问题
- Token 预算：记忆淘汰问题，用三级压缩，不是检索问题
- 长期记忆：最接近 RAG，但我没有用向量数据库——Agent 的记忆量很小，不需要近似检索。LLM 通过工具自主管理而非被动检索"

"核心区别：RAG 是**检索驱动**，我的记忆系统是 **Agent 自主驱动**——Agent 自己决定什么时候该记住什么。"

---

### Q6: 权限包装模式为什么比中间件好？

"中间件方案需要在框架层面插入逻辑——自定义 ToolNode 子类、request/response 勾子。好处是集中管控，坏处是绑定框架实现细节。langgraph 内部升级了，你的中间件可能就坏了。"

"包装模式的核心思路：我把原始工具包装成一个新的 StructuredTool，对框架来说它就是工具本身。权限检查在包装器内部完成——被拒绝时返回自然语言消息而非异常。Agent 看到'权限被拒绝，建议用 grep 代替'之后可以自己调整策略。本质是对框架的侵入程度——**组合优于继承**在工具系统上的应用。"

---

### Q7（反思）: 如果重新设计压缩系统，你会怎么改进？

"有几个方向：

第一，**职责边界**。当前 compressor 里既有纯函数（L1 压缩、token 估算）又有 LLM 调用（L2+L3）。后者应该在 token_budgeter 里——budgeter 负责决策，compressor 应该只负责纯转换。

第二，**策略插件化**。当前硬编码三级降级，但如果有人想用其他策略需要改核心代码。应该定义 `CompressionStrategy` 抽象类，token_budgeter 通过依赖注入获取策略链。

第三，**压缩效果反馈闭环**。当前压缩了一次就完了，不会评估 Agent 之后是否需要回溯旧消息。如果频繁回溯说明压太狠，下次调高阈值。静态阈值变自适应阈值。"

## 五、补充弹药

### 5.1 测试策略

- pytest-asyncio 异步测试覆盖自适应压缩全链路
- 10 个核心指标：分类 F1、Root Cause 保留率、L2 触发频率、压缩比等
- 50 样本意图分类测试（包含模糊输入），每类独立计算 F1/Precision/Recall
- 参数化压缩比测试 + L2 触发仿真
- 性能基准：L1 规则压缩 < 1ms

### 5.2 面试官可能问的"不太光彩"的问题

**Q: 你为什么不用 LangChain 自带的 memory 模块？**
A: "LangChain 的 ConversationBufferMemory 等模块设计比较重——把记忆管理和 Agent 逻辑耦合在一起。我需要三个不同层级的问题分别用不同方案解决，而不是一个统一的 Memory 对象。AsyncSqliteSaver checkpoint 做短期记忆只用了几行代码，比自己接 LangChain memory 更简单且更可控。"

**Q: 为什么不直接用 Claude Code / Copilot？你这个有什么优势？**
A: "Claude Code 是产品，我做的是系统。做这个项目的目的是理解 Agent 系统的每一层是怎么工作的——压缩、记忆、权限、纠错——做了才知道每个设计决策的 trade-off。对企业来说，他们要的不是又一个 AI 助手产品，而是能设计定制化 Agent 系统的人。"

**Q: deepseek-v4-flash 不是最好的模型，为什么选它？**
A: "正是因为不是最顶级的模型，才能暴露真正的问题。用 GPT-4 可能因为模型太强而掩盖了上下文管理等问题。用 deepseek-v4-flash 的体验更接近企业实际场景——不是每个公司都能用最贵的模型。而且它的 1M context window 是一个真实的压力测试环境。"

**Q: 你这个就是套 LangGraph 框架吧？难点在哪？**
A: "langgraph 提供了 ReAct 循环和 checkpoint，但真正的难点不在框架层。压缩系统怎么切分消息才不会拆散 tool_calls 配对？分类器怎么防止任务类型抖动？AsyncSqliteSaver 为什么 sync 调用静默失败？这些是框架不会帮你解决的。我能清楚说出哪些是框架做的、哪些是我做的、为什么这么做——这才是面试官想听的。"

### 5.3 如果面试官问"下一步做什么"

1. **压缩效果反馈闭环**：观察压缩后 Agent 是否频繁回溯旧消息，自动调高/调低阈值
2. **压缩策略插件化**：定义 `CompressionStrategy` 抽象类，支持不同策略热插拔
3. **评估框架**：场景驱动的 Agent 行为回归测试，每次改 prompt/工具描述后自动对比
4. **工具捷径学习**：Agent 犯过的错自动记录，下次相似场景注入提示
